# Python 重写版开发任务清单

## 1. `protocol` 模块

### 目标
把交易/行情/条件单的数据契约统一起来，避免业务层散落字典和临时字段。

### 任务
- 定义统一的基础枚举
  - 买卖方向
  - 开平标志
  - 价格类型
  - 有效期类型
  - 条件单状态
  - 通知等级/类型
- 定义核心数据模型
  - `Instrument`
  - `Quote`
  - `Order`
  - `Trade`
  - `Position`
  - `Account`
  - `User`
  - `Bank`
  - `TransferLog`
- 定义交易请求与回包结构
  - 登录请求
  - 下单请求
  - 撤单请求
  - 查询请求
  - 条件单请求
- 定义消息构建工具
  - `rtn_data`
  - `notify`
  - `market_quote`
  - 交易状态回包
- 定义枚举字符串/整数互转规则
- 固化 JSON 字段命名规范

### 验收标准
- 交易和行情消息都能通过同一套模型序列化/反序列化
- 不再依赖大量裸字典传递核心数据
- 所有 `aid` 消息类型都有对应结构

---

## 2. `config` 模块

### 目标
统一加载全局配置，提供所有子系统的只读配置来源。

### 任务
- 定义全局 `Config` 结构
  - 服务地址
  - 日志配置
  - CTP 配置
  - 行情配置
  - 条件单配置
  - broker 列表
- 实现配置文件加载
  - JSON
  - TOML
- 实现默认值注入
- 实现 broker 配置查询
- 实现配置路径优先级规则
- 实现配置目录与运行目录兼容
- 支持行情源类型切换
  - `tq`
  - `ctp`

### 验收标准
- 启动时可从单一配置对象完成所有初始化
- 交易、行情、条件单都能从配置中取到需要的参数
- broker 查询逻辑稳定可复用

---

## 3. `logger` 模块

### 目标
建立统一的结构化日志系统，支撑交易、行情、条件单排障。

### 任务
- 封装统一 logger 初始化
- 支持 console 输出
- 支持文件输出
- 支持日志等级控制
- 统一 `info/warn/error/debug`
- 增加上下文字段支持
  - `conn_id`
  - `symbol`
  - `order_id`
  - `user_id`
  - `aid`
- 统一异常打印格式

### 验收标准
- 各模块日志格式一致
- 关键链路可按字段过滤和定位
- 不再出现杂乱的临时打印

---

## 4. `marketfeed` 模块

### 目标
把行情侧做成独立、可复用、可切换数据源的客户端层。

### 任务
- 定义统一行情客户端接口
  - `start`
  - `subscribe`
  - `unsubscribe`
  - `get_quote`
  - `get_all_quotes`
  - `set_on_quotes`
  - `close`
- 实现 CTP 行情客户端
  - 连接前置
  - 登录
  - 订阅
  - 退订
  - 回调转发
  - 行情缓存
- 实现天勤行情客户端
- 实现连接状态查询
  - 是否已连接
  - 是否已登录
  - 当前交易日
- 实现自动重连后重新订阅
- 实现合约符号归一化
  - 支持 `SHFE.rb2610`
  - 支持 `rb2610`
- 统一 `quote` 输出格式
- 支持按连接/引擎回调推送行情

### 验收标准
- 行情源可在 `tq` 和 `ctp` 之间切换
- 订阅后可稳定收到行情推送
- 断线重连后可恢复订阅
- 行情缓存查询可用

---

## 5. `websocket` 模块

### 目标
提供稳定的连接通道，让交易/行情消息可以按连接隔离。

### 任务
- 连接建立与销毁管理
- `conn_id` 分配与维护
- 单播发送
- 广播发送
- 收消息转发到业务层
- 断开连接清理
- 连接状态维护
- 心跳/保活机制
- 消息格式统一入口

### 验收标准
- 每个连接都能被独立识别
- 消息能准确路由到对应连接
- 断线后不会残留旧连接状态

## 6. `trader` 模块

这是最重要、最复杂的一组任务，建议拆分推进。

---

### 6.1 交易主状态机

#### 目标
把交易网关的生命周期完整表达出来。

#### 任务
- 定义交易状态枚举
- 实现状态切换逻辑
- 明确状态迁移条件
- 处理停止/重连/异常退出
- 统一状态查询接口

#### 验收标准
- 登录前、登录中、登录后、重连、停止等状态清晰可追踪
- 不会出现“业务在跑但状态不一致”的情况

---

### 6.2 登录与认证链路

#### 目标
完成交易登录的完整闭环。

#### 任务
- 处理登录请求
- 读取 broker 配置
- 支持自定义 front / broker_id 覆盖
- 处理认证
- 处理用户登录
- 处理登录超时
- 处理登录失败回包
- 处理结算单查询与确认
- 处理交易日切换
- 处理重连后的恢复

#### 验收标准
- 登录成功后能进入 ready
- 登录失败有明确回包
- 超时会兜底
- 交易日变化能正确清理旧数据

---

### 6.3 用户数据模型与回包

#### 目标
建立交易侧统一的用户态数据结构。

#### 任务
- 定义用户对象
- 定义账户对象
- 定义持仓对象
- 定义订单对象
- 定义成交对象
- 定义银期转账对象
- 定义银行对象
- 实现用户数据快照输出
- 实现差量更新输出
- 实现 peek_message 返回全部用户数据

#### 验收标准
- 客户端可一次性拿到完整用户态数据
- 后续更新可以按 diff 推送
- 数据结构和 Go 侧保持一致或可映射

---

### 6.4 下单/撤单链路

#### 目标
把交易最核心的委托链路补完整。

#### 任务
- 解析下单请求
- 支持字符串和整数枚举
- 映射到 CTP 委托字段
- 管理 order_ref
- 维护本地/远端订单映射
- 处理下单响应
- 处理订单回报
- 处理成交回报
- 处理撤单请求
- 处理撤单响应
- 处理撤单错误回报

#### 验收标准
- 下单后能收到订单状态变化
- 成交回报能正确更新本地订单和持仓
- 撤单链路完整可用
- 订单 key 能稳定追踪

---

### 6.5 查询链路

#### 目标
把账户、持仓、委托、成交查询做完整。

#### 任务
- 实现账户查询
- 实现持仓查询
- 实现委托查询
- 实现成交查询
- 实现查询调度器
- 实现查询节流
- 查询完成后进入 ready
- 查询完成后触发用户数据发送
- 持仓初始化与成交重放

#### 验收标准
- 登录后能拉取完整交易态数据
- 查询不会频繁冲突
- 查询完成后客户端数据一致

---

### 6.6 转账、密码、通知类功能

#### 目标
补齐交易外围能力。

#### 任务
- 资金密码修改
- 登录密码修改
- 银期转账请求
- 转账流水查询
- 账户注册关系查询
- 交易通知接收
- 合约状态通知接收

#### 验收标准
- 各类辅助交易功能都有清晰入口
- 错误和成功通知统一输出

---

### 6.7 持久化

#### 目标
保证重连和跨交易日恢复能力。

#### 任务
- 保存订单 key 映射
- 按用户区分持久化文件
- 按交易日区分文件内容
- 重连前保存
- 重连后恢复
- 文件损坏/缺失时安全降级

#### 验收标准
- 重连后订单关联关系能恢复
- 交易日切换不会错误复用旧数据

---

### 6.8 调度与节流

#### 目标
避免查询和推送过于频繁。

#### 任务
- 查询请求节流
- 数据发送节流
- 查询请求/响应计数
- 需要查询 broker 参数的标志位
- 需要查询银行/签约关系的标志位
- 需要保存文件的标志位

#### 验收标准
- 不会出现高频重复查询
- 查询和发送逻辑可控
